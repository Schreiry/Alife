# ALife Simulation

Двумерная симуляция искусственной жизни на Python + Pygame.
Существа имеют геном из 170 параметров (физика, метаболизм,
интеллект, инстинкты, социальные качества, бой, размножение,
мутации, внешний вид). Из этих параметров вычисляются производные
характеристики, которые управляют поведением: поиском еды, бегством,
атакой, размножением, формированием кланов и территориальной
экспансией. Пользователь не управляет существами напрямую — он
наблюдает, как из набора простых правил появляется сложная динамика.

## Требования

- Windows 11 (проверено), также работает на macOS / Linux при наличии Python.
- Python 3.11 или новее, доступный в `PATH` как `python`.
- Около 200 МБ свободного места под виртуальное окружение и зависимости.

Зависимости (устанавливаются автоматически):

- `pygame-ce >= 2.5` (drop-in замена `pygame`, поддерживает Python 3.11–3.14)
- `numpy >= 1.24`
- `numba >= 0.59` — JIT-компиляция горячего кернела восприятия. Если
  Numba недоступна, используется чистый NumPy fallback (медленнее,
  но симуляция продолжает работать).

## Запуск

Самый простой способ — запустить `run.bat` двойным щелчком в проводнике.
Скрипт сам создаст виртуальное окружение `.venv`, установит зависимости
и запустит симуляцию. После выхода окно остаётся открытым (`pause`),
чтобы можно было прочитать любые сообщения об ошибках.

Альтернативно, из командной строки:

```
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
python main.py
```

## Основные механики

- **Геном**: 170 нормализованных значений 0..1, конвертируются в
  реальные диапазоны через `genetics/genes.py`.
- **Наследование**: каждый ген ребёнка — взвешенная сумма родителей,
  вес из `[0.35, 0.65]` подбирается per-gene.
- **Мутации**: вероятность и сила берутся из соответствующих генов
  родителей; гибриды получают небольшой бонус к мутациям.
- **Виды**: реестр считает дистанцию от генома существа до
  «сигнатур» видов; если расстояние > порога, заводится новый вид.
- **Восприятие**: соседи и еда находятся через spatial hash — стоимость
  на одно существо зависит только от `vision_range`.
- **Решения**: каждое действие получает score; выбирается максимум +
  небольшая случайность, масштабируемая `impulsiveness`.
- **Бой**: `damage = attack * rand - defense * mod`; победитель получает
  часть энергии жертвы и confidence.
- **Размножение**: проверки возраста, энергии, кулдауна, генетической
  совместимости и плотности; ребёнок получает кроссовер + мутации.
- **Кланы**: создаются существами с высоким `leadership` и
  `clan_creation_chance`; существа со слабой социальностью могут
  присоединяться к ближайшему клану.
- **Территории**: каждое владение клана хранится как `(owner, strength)`
  на тайл; усиливается при пребывании, медленно затухает.
- **Дипломатия**: relations ∈ `[-1, 1]`, понижаются при убийствах,
  медленно дрейфуют к 0 при отсутствии конфликта.

## Расширение генома

Каталог генов задаётся одним списком в [genetics/genes.py](genetics/genes.py).
Добавление нового гена сводится к одной строке `GeneSpec(...)`. Геном
хранится как numpy-массив фиксированной длины, поэтому `Genome` сам
проверяет размер. Производные характеристики читаются по имени:
`creature.genome.real("vision_range")`. Чтобы новое значение влияло на
поведение — добавьте поле в `Creature.attach_phenotype` и используйте
его в `behavior/decisions.py` или в одном из action-handler'ов.

## Производительность

Архитектура изначально data-oriented: горячие per-creature поля
(положение, энергия, здоровье, возраст, кэш фенотипа) живут в плотных
numpy-массивах (`CreatureStore`), а не в Python-объектах. Это даёт:

- **Numpy-counting-sort spatial grid**: rebuild O(N), query возвращает
  готовый numpy-массив индексов. Без dict-lookup в hot path.
- **Numba-кернел восприятия**: вся внутренняя петля обхода соседей
  скомпилирована в native-код. Per-creature Python overhead исчезает.
- **Vectorized batch updates**: возраст, базовый расход энергии, голод,
  старение, регенерация — одной numpy-операцией на всю популяцию.
- **Throttled subsystems**: дипломатия, статистика, decay территории,
  пересинхронизация видов — каждый со своим интервалом, см. `config.py`.
- **Decoupled main loop**: симуляция и рендер разнесены, симуляция
  может прокручиваться несколько тиков за кадр (cap `MAX_STEPS_PER_FRAME`).
- **Cached territory surface**: 1-px-per-tile карта владений
  перерисовывается только когда `world.territory_dirty`.
- **Throttled HUD**: текст overlay перестраивается раз в N кадров.

Замеренные показатели на разработческом ноутбуке (Python 3.14, world
300×300):

| Популяция | ms/tick | tps  |
| --------- | ------- | ---- |
| 200       | 3-4     | 280+ |
| 1400      | 12      | 80   |
| 2800      | 36      | 28   |
| 3000 cap  | 56-60   | 16   |

Первый тик включает JIT-компиляцию Numba — занимает около 1 секунды.
Последующие тики идут по скомпилированному коду.

## Возможные улучшения

- Замена скорингового мозга на нейросетевой (сохраняя интерфейс
  `Brain.step`).
- Доминантные/рецессивные аллели в `inheritance.py`.
- Полноценная система ландшафта (вода, биомы, погода).
- Графики по видам и кланам в боковой панели.
- Параллелизация тика по worker'ам / multiprocessing.
- Сохранение полной сетки территорий и истории генеалогий.
